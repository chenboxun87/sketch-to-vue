# MasterGo 任务提示词模板

来源：xdt-mastergo-to-frontend/prompt-templates.md（完整整合）

配合 design-to-vue skill 的 **C 轨道（MasterGo MCP）** 使用。

---

## 使用建议

发起任务前，尽量补齐以下信息：

- **目标项目**：仓库名或目标路径
- **实现范围**：整页 / 局部模块 / 单个组件
- **视觉要求**：是否要求 1:1 还原
- **改动边界**：是否允许修改页面私有组件、共享业务组件、底层组件
- **状态要求**：是否需要补齐空态、加载态、错误态、弹窗态等设计稿未覆盖的状态

---

## 模板一：只分析设计稿

适用于先做结构拆解、组件建议和实现规划，**不立即改代码**。

```text
先不要改代码。

请按 design-to-vue C 轨道的方式，分析这个 MasterGo 设计稿：
[粘贴 MasterGo 链接]

目标项目：
[项目名或路径，例如 your-web / src/pages/xxx]

请输出：
1. 页面主结构拆分
2. 哪些区域适合做成页面内模块，哪些适合抽公共组件
3. 哪些区域建议资源化处理（导出为图片），不要硬编码还原
4. 设计稿里已有的状态，以及缺失但实现时需要补齐的状态
5. 如果在这个项目里落地，优先应该复用哪些现有组件或模式
6. 建议的实现顺序

不要直接生成最终代码，也不要直接照搬 D2C。
```

## 模板二：落地到现有项目

适用于按设计稿修改已有页面或新增页面，**最常用的模板**。

```text
根据这个 MasterGo 设计稿，在指定项目里完成前端实现：
[粘贴 MasterGo 链接]

目标项目：
[项目名或路径]

实现范围：
[整页 / 某个模块 / 某个组件]

要求：
- 先分析设计结构，再开始改代码
- 优先复用项目现有组件、样式变量、布局和交互模式
- 不要直接把 D2C 结果当最终代码提交
- 如果设计稿缺少空态、加载态、错误态、弹窗态，请按当前设计语言补齐
- 不要随意改底层基础组件，除非确有必要并先说明
- 完成后给我说明：用了哪些设计信息、怎么拆分工程结构、哪些地方为了可维护性做了调整

如果信息不足，先问我，不要猜。
```

## 模板三：抽成公共组件

适用于发现同类区域会复用，希望先抽出可维护组件再集成到页面中。

```text
基于这个 MasterGo 设计稿，帮我把目标区域抽成可复用前端组件：
[粘贴 MasterGo 链接]

目标项目：
[项目名或路径]

组件范围：
[说明是哪个区域，最好带 layer 信息或页面描述]

要求：
- 先分析组件边界，再实现
- 先明确组件的 props / events / slots 或等价接口
- 优先贴合当前项目已有组件风格和命名方式
- 不要只做视觉壳子，要考虑基础态、禁用态、空态、加载态等必要状态
- 如果适合做页面私有组件，不要过度上升为全局组件
- 完成后说明这个组件为什么这样抽象，以及后续如何复用

不要机械按节点树一层层翻译成代码。
```

## 日常短版模板

适用于快速发起最常见的任务，省去详细说明：

```text
根据这个 MasterGo 链接，先分析再在 [项目名] 里落地实现。
优先复用现有组件和样式体系，不要直接照搬 D2C，缺失状态按设计语言补齐。
[粘贴 MasterGo 链接]
```

## 何时需要深挖 DSL

当出现以下情况时，可以额外要求使用 MasterGo 脚本深挖：

- 想直接查看更原始的 DSL 结构（`mastergo_get_dsl.py`）
- 想确认某个图层或区域到底是什么节点类型（`mastergo_analyze.py`）
- 需要单独抓取组件文档（`mastergo_fetch_docs.py`）
- 想核对 MCP 返回结果是否遗漏了某些设计信息

补充说法示例：

```text
这个区域我想再看一下原始 DSL 和组件文档，
请运行 mastergo_analyze.py 和 mastergo_get_dsl.py 做补充分析，
但最终实现仍按 design-to-vue 工程方式落地。
```

---

> **原则**：`mastergo_get_dsl.py` 负责"深挖 DSL + 补充证据"，C 轨道工作流负责"分析 + 工程落地"。
